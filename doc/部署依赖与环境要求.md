# 部署依赖与环境要求

## 一、本文档用途

本文档面向项目部署、运行与服务器环境准备，重点说明以下问题：

- 项目启动前需要安装哪些基础环境
- 需要哪些 Node.js 依赖
- 哪些功能依赖 Python
- 哪些功能依赖额外的系统命令行工具
- macOS 本地运行与 Linux 服务器部署有什么差异
- `api3`（Gemini）在什么网络条件下才能正常使用

如果你准备把本项目从本地开发环境迁移到服务器，这份文档应优先阅读。

---

## 二、最低运行要求

项目要正常启动，至少需要以下基础环境：

### 1. 基础运行环境

- **Node.js**
- **npm**
- **Python 3**
- **curl**

其中：

- Node.js + npm 用于启动后端与安装 Node 依赖
- Python 3 用于图形生成子系统
- `curl` 用于 Gemini provider 的原生 HTTP 请求路径

### 2. Node.js 依赖

项目 `package.json` 中声明的依赖为：

- `openai`
- `@google/genai`
- `katex`

安装方式：

```bash
npm install
```

### 3. Python 依赖

如果你要使用图形生成功能，需要安装：

- `matplotlib`

安装方式：

```bash
pip3 install matplotlib
```

### 4. 系统工具依赖

如果你要使用 PDF 相关功能，还需要：

- `pdfinfo`
- `pdftoppm`

如果你要使用当前版本的 PDF 裁图流程，还需要：

- `sips`

注意：

- `pdfinfo` / `pdftoppm` 通常来自 **Poppler** 工具集
- `sips` 通常是 **macOS 自带工具**，Linux 服务器上通常没有

## 三、依赖与功能的对应关系

这一节不是只列工具名，而是明确“哪个功能依赖什么”。

### 1. 服务启动

项目启动入口为：

- `package.json`
- `src/server/index.js`

启动命令：

```bash
npm start
```

实际执行：

```bash
node src/server/index.js
```

因此，只要要运行整个后端，就必须具备：

- Node.js
- npm

### 2. 图形功能依赖什么

图形执行器位于：

- `src/diagram/python-runner.js`

该文件中直接通过：

```js
spawn('python3', [scriptPath], ...)
```

执行 Python 脚本。

这说明：

- 系统必须能找到 `python3`
- 图形功能不是纯 Node 实现
- 运行图形功能的机器必须具备 Python 运行环境

同时，这个图形子系统的设计前提是模型生成 matplotlib 代码，因此部署环境中还需要安装：

- `matplotlib`

如果缺少这些依赖，以下功能会受到影响：

- 几何图形生成
- 数学图辅助输出
- 某些 DSE 出题流程中的自动配图
- 单独触发 `/api/diagram` 的能力

### 3. PDF 渲染依赖什么

PDF 渲染实现位于：

- `src/pdf/renderer.js`

其中会直接调用：

- `pdfinfo`
- `pdftoppm`

用途分别是：

- `pdfinfo`：读取 PDF 页面尺寸信息
- `pdftoppm`：把 PDF 页面渲染成图片

这意味着：

- 如果没装 `pdfinfo`，系统无法正确读取页面尺寸
- 如果没装 `pdftoppm`，系统无法把 PDF 页面转为图像
- 整卷 PDF 批改功能会直接受影响

### 4. PDF 裁图依赖什么

PDF 裁图实现位于：

- `src/pdf/cropper.js`

该文件会直接调用：

```js
execFile('sips', ['-c', cropArg, imagePath, '--out', outputPath], ...)
```

这说明当前版本的裁图流程依赖：

- `sips`

而 `sips` 基本是 macOS 环境中的常见工具，不是 Linux 服务器上的通用工具。

因此，这一部分要特别说明：

- 当前 PDF 裁图实现带有 **macOS 平台依赖**
- 如果部署到 Linux，不能默认认为这部分功能可以原样使用

### 5. Gemini / api3 依赖什么

`api3` 配置位于：

- `config/test-api.js`

其中 `api3` 为：

- `providerType: 'gemini'`
- `apiKey: process.env.GEMINI_API_KEY || '...'`

这说明：

- 最推荐通过环境变量设置 `GEMINI_API_KEY`
- 如果不设，当前开发配置会退回到文件中的值

真正发请求时，Gemini 路径在：

- `src/model/client.js`

该文件会通过 `curl` 直接访问 Google API，并显式带上以下代理环境变量：

- `https_proxy`
- `http_proxy`
- `all_proxy`

这意味着 Gemini 路径通常要求满足以下条件：

- 有可用的 `GEMINI_API_KEY`
- 当前运行机器能够访问 Google AI 服务
- 如果当前网络不能直连 Google API，就需要可用代理
- 系统上还要有 `curl`

## 五、macOS 本地运行与 Linux 服务器部署差异

这是部署时最需要讲清楚的一节。

## 1. macOS 本地环境

当前项目原始实现明显更接近 macOS 本地工作台环境。

优点：

- `sips` 通常已存在
- Python 安装与使用较直接
- PDF 工具可以额外安装
- 本地图形功能与裁图流程更容易按现状运行

因此：

- **macOS 是当前版本更自然的运行环境**

## 2. Linux 服务器环境

Linux 服务器通常可以安装：

- Node.js
- npm
- Python 3
- pip
- curl
- Poppler（提供 `pdfinfo` / `pdftoppm`）
- `matplotlib`

所以这些功能通常可以运行：

- 服务启动
- 常规模型调用
- DSE 出题
- 解题与批改主流程
- Python 图形生成
- PDF 页面渲染

但有一个关键限制：

- 当前版本的 PDF 裁图流程依赖 `sips`
- `sips` 不是 Linux 常规工具
- 所以 **Linux 服务器不能保证完整支持当前的 PDF 裁图路径**

结论要写得很清楚：

> 本项目可部署到 Linux 服务器，但当前 PDF 裁图实现依赖 macOS 的 `sips`。若部署环境为 Linux，基础 API、模型调用、图形生成与 PDF 渲染通常仍可运行，但涉及 `sips` 的裁图流程不能默认视为完整可用。

## 六、服务器上到底能装什么

### 1. Python 和 matplotlib

可以装。

这部分在 Linux 服务器上通常没有问题。

例如：

```bash
python3 --version
pip3 install matplotlib
```

### 2. pdfinfo 和 pdftoppm

可以装。

这两个一般通过 Poppler 安装。

例如在 Ubuntu / Debian：

```bash
sudo apt update
sudo apt install -y poppler-utils
```

安装后通常就会有：

- `pdfinfo`
- `pdftoppm`

### 3. curl

通常服务器本来就有；如果没有，也能安装。

例如：

```bash
sudo apt install -y curl
```

### 4. sips

这里要特别注意：

- 如果服务器是 **macOS**，通常可以使用 `sips`
- 如果服务器是 **Linux**，通常不能把 `sips` 当成可用前提

所以针对你的问题，最准确的说法是：

- **Python 可以装**
- **matplotlib 可以装**
- **pdfinfo / pdftoppm 可以装**
- **curl 可以装**
- **但 `sips` 不能当作 Linux 服务器上的通用依赖**

## 七、`api3` / Gemini 的启动条件

### 1. `api3` 是什么

`api3` 在当前开发配置里对应：

- Google AI Studio
- Gemini provider

### 2. 必需条件

如果你要让 `api3` 正常工作，通常要满足以下条件：

- 配置 `GEMINI_API_KEY`
- 服务器或本地机器能访问 Google API
- 如果网络不能直连 Google，则要配置代理
- 机器上要有 `curl`

### 3. 代理环境变量是什么意思

代码里会用到：

- `https_proxy`
- `http_proxy`
- `all_proxy`

这些变量的意思是：

- 把程序发出的 HTTP/HTTPS 请求交给代理转发
- 不是项目内建 VPN
- 而是使用当前机器已经配置好的代理环境

### 4. 在服务器上设置 `127.0.0.1:7890` 代表什么

这一点非常重要。

如果你在服务器上写：

```bash
export https_proxy=http://127.0.0.1:7890
export http_proxy=http://127.0.0.1:7890
export all_proxy=socks5://127.0.0.1:7890
```

这里的 `127.0.0.1` 指的是：

- **服务器自己这台机器**

不是你的电脑。

所以只有在以下情况下才有效：

- 服务器本机真的有代理程序跑在 7890 端口

如果只是你自己的电脑开了代理，而服务器没有，那么服务器里的 `127.0.0.1:7890` 是无效的。

### 5. 对 `api3` 最准确的部署结论

可以这样理解：

- 不是一定要“开 VPN”这个动作本身
- 而是 **当前运行机器必须具备能访问 Gemini API 的网络条件**
- 这个条件可以来自：
  - 机器本身能直连 Google
  - 或机器本身已正确配置代理

## 十一、最终结论

对于这个项目，最准确的部署结论如下：

1. **项目基础运行并不复杂**：只要有 Node.js、npm，就可以启动后端。
2. **图形功能需要 Python 环境**：至少要有 `python3`，并安装 `matplotlib`。
3. **PDF 功能需要额外系统工具**：至少要有 `pdfinfo` 和 `pdftoppm`。
4. **当前 PDF 裁图实现带有 macOS 依赖**：因为使用了 `sips`，所以 Linux 服务器不是完全无差别部署环境。
5. **Gemini / api3 对网络环境有要求**：除了 `GEMINI_API_KEY`，还需要当前运行机器能访问 Google API，必要时需配置代理。

如果只是问“服务器能不能跑”：

- **大部分功能可以**
- **Python、matplotlib、Poppler、curl 都可以在服务器上安装**
- **但 `sips` 相关的 PDF 裁图能力不能默认视为 Linux 可完整支持**

因此，目前最稳妥的说法是：

> 本项目可以部署到服务器，且大部分功能可运行；但若目标环境为 Linux，当前依赖 `sips` 的 PDF 裁图流程存在平台限制，若要完整覆盖该能力，后续应替换为跨平台实现。
