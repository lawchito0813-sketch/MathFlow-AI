# AI 数学运算后端开发项目文档

## 1. 项目概览

这是一个基于 Node.js 的本地 AI 数学工作台，面向多个数学场景提供统一后端与多页面前端：

- AI 解题（困难版）
- AI 解题（简单版）
- 学生作答批改（困难版）
- 学生作答批改（简单版）
- 整卷 PDF 批改
- DSE 出卷工作台

项目的核心特点不是单一模型调用，而是把多种 AI 任务组织成可追踪的工作流：

- 前端发起请求后，后端先返回 `sessionId`
- 真正的 AI 流程在后台异步运行
- 运行中的状态、流式文本、阶段切换、错误与结果通过 SSE 实时推送到前端
- 多数结构化阶段都会经过 `Prompt -> JSON 解析 -> Schema 验证 -> Repair 重试 -> Fallback` 的统一控制链路

因此，这个项目既是一个数学 AI 应用，也是一个“多工作流 + 流式事件 + 结构化输出校验”的后端实验平台。

### 1.1 相关文档

如果你是第一次接手这个项目，建议按用途阅读：

- `doc/开发者API与源码详解.md`：面向开发者，重点看 API、路由、runtime、session、SSE、Prompt/Schema 与源码结构
- `doc/项目简介.md`：面向老师、主任、校长等非开发读者，用于正式介绍项目定位与教学价值
- `doc/部署依赖与环境要求.md`：面向部署与运行环境，重点看 Node/Python/PDF/代理依赖、macOS 与 Linux 差异、Gemini 使用条件

## 2. 技术栈与运行方式

### 2.1 技术栈

- 运行时：Node.js ESM
- HTTP 服务：Node 原生 `http`
- 前端：原生 HTML + CSS + 浏览器端 ES Module
- 数学渲染：KaTeX
- 模型接入：
  - Azure OpenAI
  - OpenAI-compatible API
  - Gemini
- 图形生成：Python + matplotlib
- PDF 处理：`pdfinfo`、`pdftoppm`、`sips`

### 2.2 启动入口

项目入口定义在 `package.json:5-7`：

- `npm start`
- 实际执行：`node src/server/index.js`

服务主入口在 `src/server/index.js:1-39`：

- 初始化全局 session store
- 初始化全局 session emitter map
- 初始化 AI scheduler
- 创建 HTTP server
- 把所有请求交给 `routeRequest(...)`
- 默认监听 `3000` 端口

### 2.3 关键全局对象

`src/server/index.js:12-22` 在 `globalThis` 上挂了三个关键对象：

- `__MATH_SESSIONS__`：会话数据
- `__MATH_SESSION_EMITTERS__`：SSE 推送器
- `__AI_SCHEDULER__`：AI 请求调度器

这说明项目虽然不是数据库驱动，而是以“进程内内存状态”作为主要运行时存储。

## 5. 整体运行架构

### 5.1 请求进入后的主链路

项目的核心运行方式是：

1. 浏览器打开某个页面
2. 前端脚本请求 `/api/providers` 加载可用模型
3. 用户提交表单到某个 `/api/*` 接口
4. 后端先返回 `{ sessionId }`
5. 前端用 `/api/events?sessionId=...` 建立 SSE
6. 后端在后台启动实际工作流
7. 各阶段通过事件持续推送到前端
8. 前端按事件更新 timeline、流式输出、结果面板

### 5.2 为什么 API 会先返回 200

多个接口都采用同一模式，例如：

- `handleSolve(...)` in `src/server/routes.js:104-140`
- `handleReview(...)` in `src/server/routes.js:142-178`
- `handleDseAuthorGenerate(...)` in `src/server/routes.js:474-513`
- `handleDseAuthorFollowup(...)` in `src/server/routes.js:515-553`

它们会先：

- 校验输入
- 生成或确定 `sessionId`
- 立即 `sendJson(res, 200, { sessionId })`

然后再通过：

- `process.nextTick(...)`
- 或 `setTimeout(startFlow, 0)`

异步启动真正工作流。

这样设计的意义是：

- 前端不会被长耗时模型调用阻塞
- 页面可以立刻建立事件流
- 所有阶段状态都能逐步显示

## 6. SSE 与 Session 架构

### 6.1 SSE 实现

`src/server/sse.js:0-14` 非常简洁：

- `initSse(res)`：设置 `text/event-stream`

### 6.2 事件连接入口

`src/server/routes.js:75-102` 的 `handleEvents(req, res)`：

- 要求 `sessionId`
- 初始化 SSE 响应
- 为当前 session 注册 emitter
- 在连接关闭时删除 emitter
- 连接建立后立即发送 `session_created`

### 6.3 Session Store

`src/core/session-store.js:0-84` 负责管理内存会话：

- `setSession(sessionId, session)`
- `getSession(sessionId)`
- `getAllSessions()`
- `updateSession(...)`
- `appendFollowupMessage(...)`
- `appendSessionMessage(...)`
- `appendToolCall(...)`
- `getSessionDebugLog()`

特点：

- 全部使用内存 Map
- 会记录 debug 历史
- 没有持久化数据库
- 适合本地工作台、调试、实验流程

### 6.4 事件定义

`src/utils/events.js:0-103` 定义了全量 `SESSION_EVENT_TYPES`。

事件大致分为：

- solve 流程事件
- review 流程事件
- paper-review 流程事件

- stage repair / failed 事件

这套事件常量是连接前后端心智模型的关键，文档阅读时应把它看作“全项目事件协议”。

## 8. Structured Stage 机制

`src/core/structured-stage.js:91-234` 是整个项目中最关键的通用骨架之一。

它抽象了“结构化 AI 阶段”的统一执行方式：

1. 发送主 prompt
2. 收集模型输出文本
3. 从文本中解析 JSON
4. 用 validator 做 schema 校验
5. 如果失败，发 repair prompt
6. 如果还失败，必要时走 compact retry
7. 如果仍失败，根据配置决定 fallback 或抛错

### 8.1 它解决的问题

模型输出经常出现：

- JSON 不合法
- 缺字段
- 类型错误
- 输出被 markdown 包裹
- 内容太长或半截截断

这个模块把这些问题收敛成统一流程，避免每条业务线自己重复实现。

### 8.2 与事件系统的结合

该模块还会发出：

- `model_call_started`
- `model_call_delta`
- `model_call_done`
- `model_call_failed`
- `stage_repairing`
- `stage_compact_retry`
- `stage_failed`

所以前端能直接看到“模型调用中 / 修复中 / 保底结果生效”等状态。

- 只能输出合法 JSON
- 必须生成可执行 Python
- 只能使用标准库与 matplotlib
- 必须使用 `__OUTPUT_IMAGE_PATH__`
- 必须使用 `__FIGSIZE__`
- 不允许 `tight_layout()`
- 不允许把解题过程写到图上
- 只能输出规定结构的 JSON

### 13.3 Python 执行器

`src/diagram/python-runner.js:58-138` 会：

1. 检查 Python 代码是否包含 `__OUTPUT_IMAGE_PATH__`
2. 检查是否包含 `__FIGSIZE__`
3. 检查是否违规硬编码 figsize
4. 检查是否使用 `tight_layout()`
5. 检查是否把解题过程写进图上
6. 把占位常量替换成真实输出路径和尺寸
7. 调用 `python3` 执行
8. 读取 JPG 并返回 base64 data URL

### 13.4 这条链路的价值

这条子系统让模型不直接产图，而是：

- 先产结构化画图计划 / Python 代码
- 后端验证后再执行
- 失败时可重试修复

相比直接让模型生成图片，这种方式更可控、更可调试。

## 17. 开发与排错指南

### 17.1 新增 Provider

主要改动点：

- 开发环境：`config/test-api.js`
- 生产环境：`src/model/config.js`

如果是 OpenAI-compatible：

- 配置里写根地址，如 `https://api.example.com`
- 不要手动加 `/v1`
- 因为 `src/model/client.js:223-227` 会自动补 `/v1`

### 17.2 看一个流程具体用了哪些 Prompt / Schema

建议从 runtime 文件反查：

- solve / review / paper-review：`src/core/orchestrator.js`
- DSE：`src/core/dse-agent-runtime.js` 或 `src/core/dse-author-flow.js`

再顺着 import 看：

- `src/prompts/index.js`
- `src/schemas/index.js`

### 17.3 SSE 不更新时先检查什么

先看：

1. 前端是否先成功连接 `/api/events`
2. 当前 sessionId 是否一致
3. `globalThis.__MATH_SESSION_EMITTERS__` 是否已有该 session emitter
4. 后端是否走到了 `process.nextTick(...)` 中的流程
5. 是否触发了 `session_error`

关键文件：

- `src/server/routes.js`
- `src/server/sse.js`
- `src/core/session-store.js`
- 对应前端 `connectEvents(...)`

### 17.4 Diagram 常见失败点

关键文件：

- `src/prompts/diagram.js`
- `src/diagram/python-runner.js`
- `src/core/dse-agent-runtime.js`

常见原因：

- Python 代码缺 `__OUTPUT_IMAGE_PATH__`
- 缺 `__FIGSIZE__`
- 使用了不允许的 `tight_layout()`
- 图上写入了解题过程
- Python 执行成功但没有输出有效 JPG

### 17.5 Paper Review 常见失败点

关键文件：

- `src/pdf/renderer.js`
- `src/pdf/cropper.js`
- `src/core/orchestrator.js`
- `public/paper-review-app.js`

常见问题：

- PDF 渲染失败
- 页面尺寸识别异常
- crop hint 不命中
- question routing 不稳定
- 某一题 review 失败但整卷仍要继续汇总

### 17.6 Session / 调试信息

可通过 session 调试接口与内存日志观察：

- `getAllSessions()`
- `getSessionDebugLog()`
- paper review debug 路由

这对排查“前端没反应，但后端其实已执行”非常有帮助。

## 19. 项目当前架构总结

这个项目的本质可以概括为：

- 一个 Node.js 本地 AI 后端
- 多个数学业务页面共享同一后端
- 通过 session + SSE 组织异步工作流
- 通过 prompt + schema + repair 约束模型输出
- 通过 diagram / PDF 子系统扩展到文本之外的能力
- 通过 DSE agent runtime 把简单的表单生成升级为持续交互式工作台

如果后续继续扩展，这个架构最值得保留的核心能力有三点：

1. **结构化 stage 骨架**：可复用于更多 AI 工作流
2. **SSE + session 模式**：适合长任务与实时可视化
3. **runtime / prompt / schema 分层**：便于调试、替换和维护
